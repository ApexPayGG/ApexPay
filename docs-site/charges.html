Marketplace charges

POST /api/v1/integrations/charges tworzy charge z podziałem (split) na portfele powiązane z subkontami (ConnectedAccount). Uwierzytelnienie: wyłącznie klucz API.

Model rozliczenia

  • Kwota całkowita (amount) — integer > 0, w groszach.
  • Split — tablica splits: [{ connectedAccountId, amount }]; suma kwot w splitach nie może przekroczyć amount.
  • Prowizja platformy — różnica między amount a sumą splitów trafia na portfel integratora (platforma) jako kredyt księgowy.
  • Portfel integratora jest obciążany pełnym amount (debet).

Idempotencja

Wymagany nagłówek Idempotency-Key (string, do 256 znaków). Powtórzenie tego samego klucza w oknie 24 h zwraca 409 (np. CONFLICT / kolizja idempotencji).

Warunki subkont

Subkonto musi należeć do tego samego integratora co klucz API, mieć status ACTIVE oraz powiązany użytkownik z portfelem. W przeciwnym razie — 403 / błąd walidacji.

Przykład: dwa subkonta + reszta na platformę

Całkowicie: 100,00 PLN (10000 gr). 70 zł dla sprzedawcy A, 20 zł dla sprzedawcy B — 10 zł zostaje jako prowizja platformy (integrator).

POST /api/v1/integrations/charges 
curl -sS -X POST "https://api.apexpay.pl/api/v1/integrations/charges" \
  -H "x-api-key: apx_live_YOUR_KEY" \
  -H "Idempotency-Key: order-2026-04-16-001" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 10000,
    "currency": "PLN",
    "splits": [
      { "connectedAccountId": "ca_seller_a", "amount": 7000 },
      { "connectedAccountId": "ca_seller_b", "amount": 2000 }
    ]
  }'

Antyfraud

Przed księgowaniem wykonywana jest synchroniczna ocena. Przy BLOCKED — odpowiedź 422 z code: FRAUD_BLOCKED. Przy FLAGGED charge może zostać utworzony z opcjonalnym fraudCheckId.

Ledger (referencje)

Wpisy księgowe używają m.in. prefiksów mkt:{chargeId}:debit, mkt:…:credit:{connectedAccountId} oraz ewentualnie mkt:…:credit:platform.
