Wypłaty (payouts)

Integrator zleca wypłatę z portfela użytkownika powiązanego z subkontem (ConnectedAccount.userId). Kwoty w JSON jako integer w groszach.

Utworzenie wypłaty

POST /api/v1/integrations/payouts

Wymagany nagłówek Idempotency-Key. Przykładowe body JSON — po prawej.

Odpowiedź 201 tworzy rekord Payout ze statusem domyślnym PENDING oraz wpis outbox payout.created. Przy niewystarczających środkach na portfelu — 402.

Body

{
  "amount": 25000,
  "connectedAccountId": "CONNECTED_ACCOUNT_UUID"
}

Statusy cyklu życia

Status Znaczenie
PENDING Utworzono zlecenie; księgowanie debetu (PAYOUT_DEBIT) wykonane przy tworzeniu.
IN_TRANSIT Opcjonalny stan pośredni (np. przekazanie do PSP).
PAID Wypłata zakończona sukcesem po rozliczeniu administracyjnym.
FAILED Niepowodzenie; przy rozliczeniu możliwy zwrot na portfel (PAYOUT_REVERSAL).

Rozliczenie (admin)

Endpoint POST /api/v1/admin/payouts/:id/settle (JWT, rola ADMIN). Przykładowe body — po prawej.

Dla PAID: aktualizacja statusu, opcjonalnie pspReferenceId, webhook payout.paid. Dla FAILED: status FAILED, zwrot środków na portfel subkonta, ledger PAYOUT_REVERSAL z referenceId pout-void:{payoutId}, webhook payout.failed.

Body

{
  "status": "PAID",
  "pspReferenceId": "PSP_REF_OPTIONAL"
}
Lista i eksport GET /api/v1/integrations/payouts oraz GET /api/v1/integrations/payouts/export (CSV) — patrz docs/API.md.
