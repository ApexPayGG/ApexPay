Kody błędów

Nowsze endpointy zwracają JSON z polami error (komunikat), code (maszyna czytelna, snake_case) oraz opcjonalnie traceId do korelacji z logami serwera.

Format odpowiedzi

{
  "error": "Validation failed",
  "code": "BAD_REQUEST",
  "traceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}

Ten sam identyfikator śledzenia jest też propagowany w nagłówku odpowiedzi (jeśli włączony middleware trace) — użyj go w ticketach wsparcia.

Tabela kodów

Poniżej kategorie błędów i typowe wartości pola code w JSON. W praktyce API może zwracać m.in. BAD_REQUEST zamiast ogólnej nazwy „validation”; dla 409 często CONFLICT (w tym kolizja idempotencji).

HTTP Kategoria / code Znaczenie i działanie
400 VALIDATION_ERROR · BAD_REQUEST Niepoprawne body lub query (walidacja / Zod) — popraw schemat żądania; nie powtarzaj identycznego payloadu bez zmian.
401 UNAUTHORIZED Brak lub wygasły token / klucz — odśwież JWT lub sprawdź nagłówek x-api-key.
403 FORBIDDEN Autoryzacja OK, brak uprawnień (np. rola, cudze subkonto) — nie retry; skoryguj uprawnienia.
404 NOT_FOUND Zasób nie istnieje lub nie jest widoczny dla kontekstu — zweryfikuj ID.
409 IDEMPOTENCY_CONFLICT · CONFLICT Powtórzony Idempotency-Key, duplikat zasobu lub niedozwolony stan — użyj nowego klucza albo odczytaj istniejący rekord.
422 FRAUD_BLOCKED Transakcja zablokowana przez reguły antyfraud — nie automatycznie retry; weryfikacja manualna.
429 TOO_MANY_REQUESTS Rate limit (np. logowanie) — exponential backoff, ogranicz równoległość.
503 SERVICE_UNAVAILABLE Zależność niedostępna (np. brak konfiguracji Autopay) lub przeciążenie — retry z backoffem.

Zalecenia

  • Loguj traceId po stronie klienta przy każdym błędzie 4xx/5xx.
  • Dla 409 idempotencji — traktuj jako sygnał, że ta sama operacja była już w toku.
  • Dla 503 — krótki backoff; przy długiej niedostępności skontaktuj się z operatorem.
